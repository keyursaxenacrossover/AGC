<h2 id="planning_and_organization">Planning and Organization Tips</h2>
<p>Proper planning is crucial for a successful multi-source feedback process. Consider the following aspects:</p>
<h3 id="h_01J3DA86Y0NM41B2PA3P35YKNB">Dashboards and Reports</h3>
<ul>
<li>
<strong>Results Presentation</strong>: Plan how to present feedback results, using dashboards or reports, access levels, and availability phases.</li>
<li>
<strong>Additional Documents</strong>: Plan additional documents like general info or data privacy, access levels, and availability phases.</li>
<li>
<strong>Downloads</strong>: Organize downloads in the feedback center or as DIN A4 PDF dashboards.</li>
</ul>
<h3 id="h_01J3DA86Y0B3JKXQ9CJYCWZYXF">Role Distribution</h3>
<ul>
<li>
<strong>Defining Roles</strong>: Identify participant groups, list access rights, and allocate roles.</li>
<li>
<strong>Participant Access Rights</strong>: Plan roles' access to feedback center content, dialogs, reports, and dashboards. Determine access phases and specific permissions.</li>
</ul>
<h3 id="h_01J3DA86Y06PJX8ZBBMEJ5BJNQ">Process Progress</h3>
<ul>
<li>
<strong>Determine Feedback Statuses</strong>: Match statuses to feedback process situations and define triggers for status changes.</li>
<li>
<strong>Data Access</strong>: Specify accessible data (e.g., questionnaires, reports) for each status.</li>
<li>
<strong>Centralized Changes</strong>: Decide if status changes should be triggered centrally for all feedback.</li>
<li>
<strong>Automated Changes</strong>: Plan automated status changes and email notifications for these changes.</li>
</ul>
<h2 id="defining_reports">Defining Reports</h2>
<p>First, define the reports based on feedback results. This is essential to:</p>
<ul>
<li>Know which reports various groups will access when defining roles and rights.</li>
<li>Provide additional documents (e.g., feedback process info, privacy protection) to users from the start.</li>
</ul>

<h2 id="configuring_feedback_roles">Configuring Feedback Roles</h2>
<p>Roles such as “self-assessment”, “superior”, “direct report”, or “peer” are specified at the project's start. Each role gets a unique ID for participant allocation, e.g., during import. Roles can be labeled bilingually and have defined acronyms. Roles determine questionnaire/report access and feedback center or Data Voyager dashboard access. Define roles in Process configuration → Roles.</p>
<h3 id="h_01J3DA86Y06XBTH7R5VC6RE83Z">Overview of Existing Roles</h3>
